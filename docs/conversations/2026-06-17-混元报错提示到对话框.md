# 2026-06-17 · 混元报错时总结并提示到对话框

## 概要

- **问题**：混元3D 生成失败时，错误被两层静默吞掉——`HunyuanModelGenerator` 502 只抛状态码丢了 middleware 的人话 `message`；`SceneBuilder.buildGeneric` catch 后只 `console.warn` 就降级 lowPoly。用户在对话框只看到「场景已生成」，实际是兜底低模，完全不知情。
- **方案**：复刻现有 `onProgress` 回调通道，新增同构的 `onWarning` 通道，把混元失败的人话总结推到对话框。
- **错误总结**：新建 `summarizeHunyuanError`，按关键字把技术 message 归类成「人话 + 一句原因」（超时 / 鉴权 / Python 环境 / 兜底）。
- **降级策略调整**（用户决定）：混元失败**不再 lowPoly 兜底**，直接返回空 `THREE.Group`（不显示该对象），保留 `dslId` 以便后续编辑 diff 重建。
- 清理了 `// TODO 验证后删` 的 `hunyuan3d/diag` 临时诊断桩（middleware 端点 + SceneBuilder 两处调用）。

## 轮次 2 · 修正 LLM 在 notes 里断言「已生成」的矛盾消息

**用户**：「既然生成错误了，为什么还会返回已通过腾讯混元3D生成橘猫高精度模型，模型归一化后缩放至约0.5米（真实猫咪体长参考）。场景为草地环境，搭配自然光照。」

**助手**：定位到这条「已生成…」**不是混元返回的，是 LLM 预先写进 DSL `notes` 的**。时序：LLM 产出 DSL 时 notes 已断言「已通过混元生成」（此刻混元未跑）→ `chatStore.send` 立刻显示 → 引擎构建时混元失败才推 system 警告，于是出现矛盾消息。

根因在 [templates.ts](../../src/ai/prompt/templates.ts) 的 BASE_SYSTEM_PROMPT：用陈述语气描述 generate 机制（"触发混元生成""归一化到单位包围盒""scale 指定米数"），LLM 把机制当成已发生事实写进 notes；且只笼统说"把说明放进 notes"，没限定措辞。

改动（单文件、治本）：强化 notes 引导——只描述场景内容与布局意图，**严禁断言生成结果**（不写"已通过混元生成""已归一化到 X 米"），并解释原因（混元在引擎构建时才跑、可能失败，AI 产出 DSL 时无法预知）。这样失败时：notes 说"放一只橘猫（混元高精度）"=意图，system 警告说"橘猫混元失败已跳过"=结果，两条不再矛盾。

**结果**：
- `npm run typecheck` 通过。
- prompt 为前端代码，HMR 生效，无需重启。
- 待运行时验证：浏览器重新生成橘猫场景，确认 notes 改为意图性措辞（不再出现"已通过混元3D生成"），失败时与红色 system 警告不冲突。
- 局限：依赖 GLM 服从 prompt 约束；若仍偶尔断言，后续可在 `chatStore.send` 显示 notes 前加引擎侧兜底（如本轮有混元失败则弱化断言性文案）。

## 轮次 3 · GLM 流式输出「思考过程」到对话框

**用户**：「调用后端模型的时候，能把思考过程一起发出来吗？」

**助手**：确认当前 [GlmAdapter](../../src/ai/adapter/GlmAdapter.ts) 非流式（`stream:false`）、不解析 reasoning。用户选定：流式实时 + 可折叠「思考」块。智谱 GLM 推理模型思维链在流式 `delta.reasoning_content`（WebSearch 确认格式，并对 `reasoning` 字段双容错）。

改动文件：
- [types.ts](../../src/ai/types.ts) —— `GenerateOptions` 加 `onReasoning?: (delta) => void`。
- [GlmAdapter.ts](../../src/ai/adapter/GlmAdapter.ts) —— `stream:true`，新增 `consumeStream`/`parseSseFrame`/`extractDelta`：按 `\n\n` 分帧，`delta.reasoning_content`（容错 `reasoning`）→ `onReasoning` 实时回调，`delta.content` 累积后 `parseJSONObject` 返回（**返回契约不变**）。`[DONE]`/流结束即停。
- [EditorEngine.ts](../../src/core/engine/EditorEngine.ts) / [editorStore.ts](../../src/ui/stores/editorStore.ts) —— `generate` 加 `opts?` 参数透传 `onReasoning`（orchestrator 已透传 opts，无需改）。
- [chatStore.ts](../../src/ui/stores/chatStore.ts) —— `ChatMessage` 加 `kind`/`thinking`；新增 `startThinking`/`appendThinking`/`finishThinking`；`send` 流程插入思考块（无内容则自动移除）。
- [ChatPanel.vue](../../src/ui/components/chat/ChatPanel.vue) —— `thinking` kind 渲染 `<details open>` 可折叠块（思考中…/思考过程），浅灰样式 + 自定义展开标记。
- [summarizeError.ts](../../src/core/capabilities/hunyuan/summarizeError.ts) —— 据 dev server 日志真实失败 message（`CREDENTIALS_NOT_CONFIGURED`），补 `credentials`/`secret_key` 关键字归类为「服务鉴权失败」。

**结果**：
- `npm run typecheck` 通过；HMR 正常。
- **附带发现**：dev server 日志显示用户实际触发的混元失败是 `CREDENTIALS_NOT_CONFIGURED`（`.env.local` 未配 `TENCENTCLOUD_SECRET_ID/KEY`），已据此完善错误归类。
- 待浏览器实测：发指令应见可折叠「思考中…」块实时填充，结束后变「思考过程」+ AI 回复 + 场景生成。若 `glm-5.1` 非推理模型（无 `reasoning_content`）则思考块不出现、DSL 照常（容错）。
