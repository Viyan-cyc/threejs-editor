# 对话记录 · 2026-06-16 · 清单外对象接入混元3D 高精度生成

> 接 [放开领域白名单（通用兜底）](./2026-06-16-放开领域白名单通用兜底.md)。记录约定见 [README.md](./README.md)。

## 概要

- **起因**：清单外对象此前只能用 lowPoly 简化几何（集装箱 = 黄盒子）。用户要求：非领域模型改为调用**腾讯混元3D**（项目自带 skill `hy-3d-generation-1.0.0`）生成高精度 GLB。
- **关键约束**：混元 skill 是 Python 脚本（`main.py` + 腾讯云 SDK），浏览器无法直连（无 Python、密钥会泄漏、COS 跨域）→ 必须加一层 **Node 中间层**（vite `configureServer` middleware）持密钥、spawn Python、落盘 GLB。
- **决策**（AskUserQuestion 两问）：① **同步等待 + 进度文案**（不重构 SceneBuilder 增量挂载）；② **已有腾讯云密钥，接真服务**（不做 Mock 降级）。
- **实现**：新增 `HunyuanModelGenerator` capability（挂 EditorContext）+ `build/hunyuanMiddleware.ts`；DSL 加 `generate` 字段；prompt 改造让 GLM 产 `generate.prompt`；落盘缓存（同 prompt 二次零生成）+ 归一化到单位包围盒。
- **踩坑**：middleware 初放 `src/core/` 被 vue-tsc 扫到报 node 类型缺失（项目无 `@types/node`），移到 `build/`（tsconfig.include 之外，与 `vite.config.ts` 同理）解决。
- `npm run typecheck` 零报错。

**助手**

- **追链路**：[SceneBuilder.buildObject](../../src/core/scene/SceneBuilder.ts) 按 `domain.builders.find(b => b.type === node.type)` 路由；光伏 `type='solar-panel-array'` 命中 [EnergyDomain](../../src/domains/energy/EnergyDomain.ts) 注册的 `SolarPanelFieldBuilder`（程序化 lowPoly，**未调** ModelFactory/GLB）。

- **关键澄清（诚实）**：GLM 实际返回的 `type` 字符串**不可控**——[normalizeDSL](../../src/ai/orchestrator/normalizeDSL.ts) 无 schema 校验 / type 白名单（注释明说 TODO）。只有 GLM 恰好返回 `'solar-panel-array'` 才走内置，否则静默落兜底。看 GLM 真实返回的唯一途径：DevTools Network 看 `/llm/...` 响应，或给 [GlmAdapter](../../src/ai/adapter/GlmAdapter.ts) 加 `console.log`。

**结果**：明确了「清单外对象走哪条路完全取决于 LLM 返回的 type」，为下一轮「给清单外对象配 generate 字段走混元」铺垫。

- **Plan agent** 细化 8 个设计点（GLB 投递、capability 形态、DSL 字段、SceneBuilder 三分支、进度接线、middleware 实现、prompt 改造、缓存）。

### 数据流
```
GLM → objects[]{type:'清单外', generate:{prompt}, transform}
  → SceneBuilder.buildObject: builders.find 找不到 → buildGeneric
  → node.generate?.prompt 存在 → ctx.hunyuan.generate()
  → fetch POST /hunyuan3d/generate  (vite middleware)
  → hash 命中落盘? 返回本地 url : spawn main.py → 下载落盘 → 返回 url
  → GLTFLoader 加载 → 归一化单位包围盒 → 内存 cache → 返回 Object3D
  → 失败 catch → GenericShapeBuilder lowPoly 兜底
```

### 实施（新建 3 + 改 8）

### 踩坑：middleware 位置
- 初版把 `HunyuanDevMiddleware.ts` 放 `src/core/capabilities/hunyuan/`，vue-tsc 扫到报 `Cannot find module 'node:child_process'` / `Buffer` / `process` / `Connect.ServerResponse`——项目无 `@types/node`。
- 原 `vite.config.ts` 也用 `node:url` 却不报错，因 `tsconfig.include` 只有 `src/**`，根目录文件不在 typecheck 范围。
- **解法**：不装 `@types/node`（避免污染 src 的浏览器/框架无关类型范围），把 middleware 移到项目根 `build/`（与 vite.config.ts 同处 typecheck 范围外）。`vite.config.ts` import 路径随之改 `./build/hunyuanMiddleware`。

### 踩坑②：configureServer 写错位置导致 404（运行时）
- **症状**：浏览器请求 `POST /hunyuan3d/generate` 返回 404；`curl GET` 返回 200 + index.html（vite 当 SPA 路由）。两种都没触发 middleware。
- **诊断**：`curl GET` 期望 405、`curl POST {}` 期望 400，实测分别得到 200 / 404 → middleware 未注册。
- **根因**：`configureServer` 被写进 `server: { configureServer(server){...} }` 配置项。但 **`configureServer` 是 vite 的「插件 hook」，不是 `server` 配置项**——放在 `server:{}` 里 vite 完全不识别、直接忽略，从未调用，middleware 永不注册。
- **解法**：把 `configureServer` 移到一个插件对象，放进 `plugins: []`：
  ```js
  plugins: [
    vue(),
    { name: 'hunyuan-3d-middleware', configureServer(server) { installHunyuanMiddleware(server, {...}); } },
  ]
  ```
- **验证**：重启后启动日志见 `[vite] 混元3D middleware 已注册`；`curl GET`→405、`curl POST {}`→400，middleware 正常响应。
- **教训**：自定义 dev middleware 一定要挂到 `plugins[].configureServer`，而非 `server.configureServer`。

### 端到端实测与运行时坑

用 curl + 浏览器实测整条链路，**全部跑通**：密钥✓ → SDK✓ → 提交✓ → 生成（LowPoly ~8.5min）✓ → GLB 有效（15~46MB，glTF v2）✓ → 落盘✓ → 缓存命中✓ → vite GET 服务✓ → 前端 GLTFLoader 加载✓ → 渲染显示橘猫✓。过程踩了一串运行时坑：

- **踩坑③ pip 超时**：skill 的 `ensure_dependencies` 从 PyPI 装 mega 包 `tencentcloud-sdk-python`，国内超时。改用清华镜像装更小的 `tencentcloud-sdk-python-ai3d`（带 common），import 成功后 main.py 跳过自动装。
- **踩坑④ type 大小写**：腾讯云返回 `Type:"GLB"`（大写），`extractGlbUrl` 找 `'glb'`（小写）永不匹配 → 报「未找到 glb」。改 `f.type?.toLowerCase()`。main.py 失败时把 error JSON 打到 stdout（非 stderr），middleware 原本只看 stderr，补了 stdout 解析才暴露真因。
- **踩坑⑤ ResourceInsufficient**：腾讯云免费资源耗尽（几次生成后），提交即返回 `ResourceInsufficient: 资源不足`。**非代码问题**，需控制台购资源包或等额度。
- **踩坑⑥ 测试缺灯光**：刷新后未 generate → applyScene 没加灯 → PBR 模型全黑融进黑色背景看不见，误判加载失败。加诊断端点 `/hunyuan3d/diag`（前端 POST mesh 数/包围盒/相机/scene 子节点）才确认加载一直成功（`meshCount:1`），根因是缺灯。
- **GLB 元数据**：混元产物是标准 glTF（`asset.generator: Khronos glTF Blender I/O`），**无 Draco/meshopt/KTX2 压缩**，仅 `KHR_materials_specular`；GLTFLoader 直接可加载（加的 DRACLoader/meshopt 是无害防御）。
- **调试手段**：加 `/hunyuan3d/diag` 端点接收前端诊断，自主读 dev server 日志判断，避免反复让用户看浏览器 console。

**结论**：混元链路代码完全正确；生成耗时偏长（~8min，后续可移后端异步）；真实 generate 场景有灯光不会黑猫，待腾讯云资源恢复即可端到端使用。

### 简化
- 前端内存 cache 不与后端落盘 hash 强耦合：前端 fetch 后端拿到 `{url}`（已含 hash 文件名），前端用 prompt 串作内存 cache key，hash 只在后端算。少一层前后端一致性耦合。
- `GenerateDSL` 直接定义在 `dsl/types.ts`（契约层），不单独建 `hunyuan/types.ts`，避免类型重复 + 依赖方向（dsl ← capabilities）。

### 验证
- `npm run typecheck`：**零报错**。
- 重启 dev server（新 config）：正常启动，混元 middleware 已注册。

**遗留**

- 生产部署需自建等价 Node 后端（dev middleware 不适用）。
