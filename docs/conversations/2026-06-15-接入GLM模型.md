## 轮次 1 · 用 GLM 替换 MockLLMAdapter

**用户**
> 我现在要接入glm模型，怎么弄

**助手**

### 调研（AI 层契约 + GLM API + CORS）
- 契约：[LLMAdapter.generateScene](../../src/ai/adapter/LLMAdapter.ts) `(GenerationPrompt, opts?) => Promise<SceneDSL>`；`BASE_SYSTEM_PROMPT` 已要求模型"仅输出合法 JSON"；唯一替换点 [EditorEngine.ts:82](../../src/core/engine/EditorEngine.ts#L82) `new MockLLMAdapter()`；`appConfig.defaultAdapter` 是死字段。
- GLM API：端点 `https://open.bigmodel.cn/api/paas/v4/chat/completions`，`Authorization: Bearer {key}`，OpenAI 兼容请求/响应。当前模型 ID `glm-4.5/4.6/4.7`（GLM-5 已上线）。
- **CORS**：浏览器直连必被拦截 → 必须 proxy。

### 实施（9 处改动）
1. [extractJson.ts](../../src/ai/adapter/extractJson.ts)：`parseSceneDSL` —— 直解 → 剥 markdown 围栏 → 截首尾 `{}`，全失败抛错。
2. [GlmAdapter.ts](../../src/ai/adapter/GlmAdapter.ts)：构造校验 key；fetch proxy 路径，非 2xx 抛带状态码的错，取 `choices[0].message.content` 交 `parseSceneDSL`。
3. [adapter/index.ts](../../src/ai/adapter/index.ts)：`createLLMAdapter({adapter, glm})` 工厂。
4. [app.config.ts](../../src/config/app.config.ts)：加 `llm: { adapter:'glm', glm:{endpoint:'/llm/...', model, apiKey, temperature} }`，删死字段 `defaultAdapter`。
5. [env.d.ts](../../env.d.ts)：`VITE_GLM_API_KEY` / `VITE_GLM_MODEL` 类型。
6. [vite.config.ts](../../vite.config.ts)：`server.proxy['/llm']` → `https://open.bigmodel.cn`，`changeOrigin`+rewrite 去 `/llm` 前缀。
7. [EditorEngine.ts](../../src/core/engine/EditorEngine.ts)：`new MockLLMAdapter()` → `createLLMAdapter(appConfig.llm)`。
8. [.gitignore](../../.gitignore)：补 `.env` / `.env.*`，并用 `!.env.*.example` 否定保留示例文件（避免误伤 `.env.local.example`）。
9. 新建 [.env.local.example](../../.env.local.example)：填写指引（key 申请地址 + 模型覆盖说明）。

### 关键决策
- **proxy 而非后端**：纯前端项目里绕 CORS 的唯一手段；dev 有效，生产需自建代理。
- **前端持 key**：本地工具可接受；公网部署必须改后端持钥，已记入文档。
- **零依赖 fetch**：不引入 SDK。
- **失败即抛错**：[editorStore.generate](../../src/ui/stores/editorStore.ts) 已 try/catch 上报 `error.value`；切回 mock 只改 `appConfig.llm.adapter='mock'`。
- 默认模型 `glm-4.6`（搜索中文档引用最稳），`VITE_GLM_MODEL` 可覆盖。

### 验证
- `npm run typecheck`：**零报错**。
- 待人工：复制 `.env.local.example` → `.env.local` 填 key → `npm run dev` → 选能源领域输入描述 → 应返回按描述变化的 DSL（区别于 mock 固定示例）。

**结果**：GLM 接入完成，Mock 保留可切。代码层面就绪，**只需用户在 [.env.local](../../.env.local.example) 填入真实 API Key 即可联调**。

**遗留说明**
- 生产部署：vite proxy 失效，需自建后端代理并把 `appConfig.llm.glm.endpoint` 指向它（同时把 key 移到服务端）。
- DSL 运行时校验：轮次 2 已补轻量 normalize（字段类型兜底）；更严格的 schema / 对象 type 白名单仍为后续 TODO。

## 轮次 2 · 修复 notes.join 报错

**用户**
> dsl?.notes?.join is not a function 报错

**助手**
- **根因**：GLM 未遵守 `notes: string[]` 契约（返回了字符串），而 [parseSceneDSL](../../src/ai/adapter/extractJson.ts) 只提取 JSON 不校验字段类型，[orchestrator](../../src/ai/orchestrator/SceneGenerationOrchestrator.ts) 的 `// TODO: validate(dsl)` 又一直没实现 → [chatStore.ts:38](../../src/ui/stores/chatStore.ts) `dsl.notes.join('\n')` 对字符串调用就炸了。
- **治本修复**：新建 [normalizeDSL.ts](../../src/ai/orchestrator/normalizeDSL.ts) `normalizeSceneDSL(dsl, domain)`——version/domain 缺失补默认、`notes` 规范为 `string[]`、`objects`/`lights` 非数组则置空；orchestrator 出口调用它（补上 validate TODO 的第一步）。
- **顺带修** [chatStore.send](../../src/ui/stores/chatStore.ts)：`??` 改 `||`（空 notes 不再显示空消息）；并修复「生成失败被掩盖」——`editor.generate()` 内部已吞错返回 null，原 catch 抓不到，现 dsl 为 null 时改显示 `editor.error` 的真实原因（联调 key/限流问题时能直接看到）。
- 注：`SceneDSL.notes` 类型仍声明为可选，chatStore 保留 `?.`（类型安全），normalize 是运行时保证。

**结果**：`npm run typecheck` 零报错。GLM 返回任意 notes 形态都不再崩，失败也能在聊天区看到真实错误。
